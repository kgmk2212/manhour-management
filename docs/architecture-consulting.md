# アーキテクチャコンサルティング — 大規模リファクタリングを視野に入れた理想形

**第2版（2026-07-06 改訂）**。初版の「個人ツール・本人のみ」前提をヒアリングで更新した:
①マルチデバイス同期は実施したい（現行の OneDrive 経由 JSON 手動共有から移行）、
②**将来は社内共有＝チームメンバーが入力し集計するマルチユーザー構想**（まずは自分だけ入力でも可）、
③社内サーバーは**外部公開不可・Tailscale も基本なし**（イントラ内完結が前提）、
④UIフレームワークにこだわりなし。
コード内の改善17件は [refactoring-proposals.md](refactoring-proposals.md)、現状の怪しい挙動は [known-behaviors.md](known-behaviors.md) を参照。

## 0. 前提の整理

| 項目 | Stage 1（いま〜個人利用） | Stage 2（社内共有） |
|---|---|---|
| 利用者 | 本人のみ・複数端末（Mac/iPhone） | チームメンバー数名〜（各自入力・集計閲覧） |
| ネットワーク | 自宅 tailnet（Mac mini 常時起動） | **社内イントラのみ・外部公開不可** |
| 認証 | 実質不要（簡易トークンで可） | 必要（社内アカウント。M365 利用中なら Entra ID が自然） |
| コスト | 課金ゼロ厳守 | 社内リソース内で完結（クラウド課金なし） |
| 現行データ共有 | JSONエクスポート→OneDrive→手動インポート | — |

**設計の核心**: Stage 1 と Stage 2 を**同一コードベース・同一アーキテクチャ**で通す。個人版を「シングルユーザーの特殊形」として作れば、社内版は「デプロイ先と認証の差し替え」になる。逆に個人版を localStorage 発想のまま作り込むと、社内版で全面作り直しになる。

---

## 1. 診断（初版から変わらない部分）

1. 最大の構造的制約は **localStorage 依存のデータ層**（単一ブラウザ・5MB・スキーマ移行なし）。マルチデバイス・マルチユーザーのどちらの要求もここで詰まる
2. 2番目は HTML インラインハンドラ88件 → `window.*` 380件 → ESM の三層結合
3. OneDrive 手動同期は「最後に書いた人が勝つ」以前の問題として、**インポート忘れ・二重編集で黙ってデータが分岐する**構造。移行優先度は高い
4. 「素のJS・ビルドなし」思想自体は美徳だが、マルチユーザー構想が出た時点で**サーバーを持つ設計は不可避**。ならば型とビルドの導入価値も跳ね上がる

## 2. 理想アーキテクチャ（目標像）

**「ローカルファースト・クライアント ＋ 同期API サーバー」の単一構成。**

```
[ブラウザ（Mac / iPhone / 社内PC）]
┌────────────────────────────────────────────┐
│ UI 層: React コンポーネント（画面単位）          │
├────────────────────────────────────────────┤
│ アプリ状態層: TanStack Query（サーバー状態）      │
│              + 軽量ローカル状態                 │
├────────────────────────────────────────────┤
│ ドメイン層: 純粋関数 TypeScript                 │ ← detectDiff/calculateProgress/
│   （DOM・ストレージ・通信に非依存）              │    営業日・丸め等。最重要資産
├────────────────────────────────────────────┤
│ 永続化層: Repository インターフェース            │
│   ・IndexedDB（ローカルキャッシュ＝オフライン可） │
│   ・同期クライアント（差分push/pull・再送キュー）  │
└──────────────────┬─────────────────────────┘
                   │ HTTPS(イントラ内) / 同一オリジン
┌──────────────────┴─────────────────────────┐
│ 同期APIサーバー（Node + Hono + SQLite）         │
│  ・静的フロントも同居配信（CORS問題を作らない）    │
│  ・認証アダプタ: Stage1=固定トークン /            │
│                Stage2=Entra ID(OIDC)差替え     │
│  ・夜間JSONエクスポート（OneDrive置き場に継続保存  │
│    ＝現行運用がそのままバックアップ経路に昇格）     │
└────────────────────────────────────────────┘
   Stage 1: Mac mini（tailnet内）
   Stage 2: 社内サーバー/VM（イントラ限定・外部公開なし）
```

### ローカルファーストにする理由

- 社内版はイントラ限定＝**社外・移動中はサーバーに届かない**。IndexedDB キャッシュ＋オフライン書き込みキューがあれば、外で入力→社内ネット到達時に自動同期、が成立する
- 個人版でも同じ理由で有効（Mac mini 停止時・圏外時）
- 「サーバー死んでもアプリは動く・戻ったら追いつく」は現行のスタンドアロン気質を損なわない

### 同期方式は「作り込みすぎない」

CRDT（Yjs等）は不採用。工数データは**行単位・ほぼ自分の行しか書かない**ワークロードなので:
- 各レコードに `id / updatedAt / updatedBy / deletedAt(tombstone) / rev(サーバー採番)` を持たせる
- クライアントは `since=rev` で差分 pull、ローカル変更を push（Last-Write-Wins）
- 衝突が現実的に起きるのは「同一レコードを2端末で同時編集」のみ → LWW＋履歴（既存 undo/redo 資産）で十分
- **この5フィールドを Phase 2（スキーマ型定義）の時点で入れておくことが、マルチユーザー対応の実質的な先行投資のすべて**

### マルチユーザー化で増えるもの（Stage 2 の差分）

| 項目 | 内容 |
|---|---|
| 認証 | 認証アダプタの差し替え（M365 環境なら Entra ID の OIDC。イントラ内でも可）。Stage 1 から「認証層を通ってから Repository に触る」形にしておく |
| データ帰属 | `updatedBy` に加え、実績レコードの「担当者」を**入力ユーザーと紐付け**（現在は文字列の担当者名 → ユーザーIDとの対応表を導入） |
| 権限 | 最小構成: 全員が全件閲覧・自分の実績のみ編集・見積/スケジュールは編集可能ロール。ロールは users テーブルの1カラムで足りる |
| 集計 | 既存レポートがほぼそのまま「チーム集計」になる（メンバー別フィルタは現UIに既にある） |

## 3. 技術スタック推奨（確定案）

フレームワークにこだわり無し＋社内共有（将来、他の人が保守・IT部門が確認する可能性）を踏まえ、**初版の Svelte 推奨から React に変更**する。理由: 社内ツールとしての引き継ぎ耐性・情報量・レビュー通過のしやすさで最も無難。移行コスト自体は両者で大差ない。

| 領域 | 推奨 | 備考 |
|---|---|---|
| 言語 | **TypeScript**（フロント/サーバー共通） | スキーマ型（Zod でバリデーション兼用）を共有パッケージ化 |
| フロント | **React 19 + Vite（SPA）** | SSR不要。ルーティングは TanStack Router か react-router の軽い方 |
| サーバー状態 | **TanStack Query** | 差分同期・オフライン再送・キャッシュの土台 |
| ローカル永続 | **IndexedDB（Dexie）** | localStorage から初回移行（旧データはバックアップキーで温存） |
| サーバー | **Node + Hono + SQLite（better-sqlite3 / Drizzle）** | 単一プロセス・単一ファイルDB。**Windows サーバーでもそのまま動く**（社内が Windows でも Node サービス化<nssm等>か Docker で可）。イントラ限定前提なので構成は最小でよい |
| 認証 | Stage1: 固定トークン → Stage2: **OIDC（Entra ID）** | 認証をミドルウェア1枚に隔離しておく |
| テスト | **Vitest**（ドメイン・同期ロジック）＋ **Playwright**（主要3フロー） | 既存 node:test 48件は機械移植可 |
| Excel/祝日 | 現行踏襲（xlsx / japanese-holidays を npm 化） | チーム集計のExcel出力は社内共有で価値が上がる |
| LLM分析 | Ollama は**個人利用機能として温存**（接続先を設定値化） | 社内版では設定でオフ（社内データを個人Macに送らない・IT説明も不要になる） |

### 選ばない選択肢（初版から追記あり）

- **クラウドDB/BaaS（Supabase/Firebase）**: 課金ゼロ主義に加え、社内版が「外部公開不可」なので**そもそも使えない**。イントラ内 SQLite が正解
- **M365 プラットフォーム内製（SharePoint リスト + Power Apps）**: 社内ホスティングが完全に不可能な場合の代替にはなるが、既存24k行の資産（ガント・レポート・Excel入出力・undo）の大半を捨てて作り直しになる。**社内サーバーに置ける見込みがあるなら選ばない**。完全に不可と判明した時のみ再検討
- **CRDT・リアルタイム共同編集基盤**: ワークロードに対して過剰
- **ゼロベース書き直し**: 引き続き強く非推奨。§5 の段階移行で到達する

## 4. デプロイ計画

**「静的フロント＋APIを同一サーバーから配る」単一デプロイ物**に収束させる（CORS・接続先設定の問題が消える）。

| 段階 | 配信 | 用途 |
|---|---|---|
| Stage 0（現状） | GitHub Pages（PUBLIC） | 継続。ただし**役割を「デモ/プレビュー」に格下げ**していく |
| Stage 1 | **Mac mini**（launchd 常駐・tailnet 内・news と同居） | 個人本番。Mac/iPhone のマルチデバイス同期が成立 |
| Stage 2 | **社内サーバー/VM**（イントラ限定・外部公開なし） | 同じ成果物を配置先だけ変えてデプロイ。Node を Windows サービス化 or Docker |

- Stage 1→2 でコードは変わらない（設定: DBパス・認証モード・ベースURLのみ）
- 社内版のTLS: イントラ限定なら社内CA or HTTP（IT方針に従う）。ここは交渉事項であり技術では決められない
- **GitHub Pages（PUBLICリポ）の扱いは Stage 2 までに要判断**: 社内業務ロジック（工程名・集計方法）が公開リポにあることを会社がどう見るか。無難なのは private リポ化＋Pages 廃止（デモは同サーバーの別パスで配る）

## 5. OneDrive 手動同期からの移行手順（データを1件も落とさない）

1. **現行フローで最新の JSON エクスポートを取る**（これが正本）
2. Phase 3（Repository + IndexedDB）完成時: そのJSONをインポート→IndexedDB へ（既存インポート機能を流用。localStorage 由来データとの重複は既存IDで排除）
3. Phase 6（同期API）完成時: IndexedDB の全量を初回 push → サーバーが正本になる
4. **以後の OneDrive の役割は「自動バックアップ置き場」に変更**: サーバーが夜間に全量JSONを書き出し、OneDrive 同期フォルダに置く（現行の手動作業が自動化された形で残る＝ロールバック手段・可搬性の保険・「OneDriveにあれば安心」という現運用の心理的連続性）
5. 手動インポート機能は**削除しない**（災害復旧経路として温存）

## 6. 移行ロードマップ（第2版）

初版の Phase 0〜4 は不変。Phase 5 以降をマルチユーザー構想に合わせて更新。

| Phase | 内容 | 得られる価値 |
|---|---|---|
| 0 | refactoring-proposals P1（整地） | 済んでいる特性テスト48件が守る |
| 1 | Vite + Vitest + TypeScript(checkJs) 導入（挙動不変） | 型検査・ビルド基盤 |
| 2 | ドメイン層抽出 + **スキーマ型定義（この時点で updatedAt/updatedBy/deletedAt/rev を入れる）** | 資産の恒久化・**マルチユーザーの先行投資はここで完了** |
| 3 | Repository 化 + IndexedDB 移行（localStorage から自動移行） | 容量・移行問題の解消 |
| 4 | UI の画面単位 React 化（依存の少ない画面から） | 巨大ファイル解体・window結合の撤去 |
| 5 | 同期APIサーバー（Mac mini）＋差分同期＋夜間OneDriveバックアップ | **マルチデバイス同期の完成・手動JSON往復の廃止** |
| 6 | 認証アダプタ実装（固定トークン運用開始） | Stage 2 への接続点 |
| 7 | 社内サーバーへデプロイ・Entra ID 接続・担当者↔ユーザー紐付け・権限1段 | **社内共有（チーム入力→集計）** |

- Phase 5 を Phase 4（UI移行）の完了前に前倒しすることも可能（Repository の裏に同期を差すだけなので、旧UIのままでも同期は動く）。**「手動JSON往復を早く卒業したい」なら 0→1→2→3→5 の順を推奨**
- Phase 7 は社内の話が具体化してから。ただし Phase 2 の5フィールドだけは先に入れる（後から全レコードに遡って足すのは痛い）

## 7. 決めたこと・残る分岐点

**決定（ヒアリング反映済み）**: マルチデバイス同期は実施する／マルチユーザーを構想に含める／外部公開なし前提／フレームワークは React（お任せ→社内引き継ぎ耐性で選定）。

**残る分岐点**:
1. **Phase 5 の前倒し可否** — 手動JSON往復の苦痛度で決める（推奨: 前倒し）
2. **社内ホスティングの実際**（Windowsサーバー？Docker可？）— Phase 7 着手前に要確認。ここが完全NGで M365 しか無い場合のみ §3 の Power Apps 代替を再検討
3. **PUBLIC リポの扱い** — Stage 2 までに private 化するか
4. **担当者マスタの持ち方** — 現在の自由文字列の担当者名を、いつユーザーIDに正規化するか（Phase 2 で対応表だけ作っておくのが安い）

## 8. まとめ（1行版）

**「React + TypeScript + Vite のローカルファーストSPA ＋ Node/Hono/SQLite の同期API」を単一デプロイ物として作り、Mac mini（個人・tailnet）→社内サーバー（イントラ限定）へ同じものを配置替えする。マルチユーザーの布石はスキーマの5フィールドで先に打ち、OneDrive は手動同期から自動バックアップ置き場へ役割変更する。**
