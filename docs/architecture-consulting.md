# アーキテクチャコンサルティング — 大規模リファクタリングを視野に入れた場合の理想形

2026-07-06 作成。[refactoring-proposals.md](refactoring-proposals.md)（コード内の改善17件）の上位文書として、**アーキテクチャ・技術スタック・デプロイまで含めた「理想形」と、そこへ安全に到達する道筋**を提案する。

## 0. 前提の明示（この提案が依拠する事実・仮定）

| 項目 | 内容 |
|---|---|
| 利用者 | 本人のみ（個人ツール）。マルチテナント・認証は不要 |
| 開発体制 | 本人＋Claude Code。**型とテストはLLM協働開発の生産性を直接押し上げる** |
| コスト方針 | 課金ゼロ主義（news プロジェクトと同じ）。クラウド有料サービスは選択肢から除外 |
| 既存資産 | Mac mini 常時起動・Tailscale tailnet・launchd 運用ノウハウ・Ollama 常駐（brew services） |
| 現デプロイ | GitHub Pages（**リポジトリはPUBLIC**・2026-07-06 実機確認）。ブランチ別プレビュー配信 |
| データ | localStorage のみ（≒5MB上限・単一ブラウザに閉じる）。JSON/Excel エクスポートあり |
| 現スタック | 素のESM JS 24k行・ビルドなし・`lib/` にライブラリ同梱（CDN非依存は美点） |

仮定が違う場合（例: 将来他人にも使わせたい）は結論が変わるので、その時は読み直しではなく再設計判断を。

---

## 1. 本質的な診断 — 問題は「フレームワークが無いこと」ではない

コード調査を踏まえた率直な診断:

1. **最大の構造的制約はデータ層**。localStorage は (a) 単一ブラウザに閉じる＝iPhone と Mac で別データ、(b) 実質5MB、(c) 同期的アクセスのみ、(d) スキーマ移行の仕組みなし。工数データは蓄積型なので、この制約はいずれ必ず当たる
2. **2番目は結合の形**。HTML インラインハンドラ88件 → `window.*` 380件 → ESM という三層結合が、分割・テスト・型付けのすべてを阻んでいる（refactoring-proposals P2-4）
3. **3番目はプライバシーの歪み**。個人の工数実績（働き方そのもの）を扱うアプリの**コードと配信が PUBLIC リポジトリ**にある。データ自体は端末内に留まるので漏れてはいないが、「アプリは公開・データは端末」という構成は本人の意図として整理しておく価値がある
4. 一方で「素のJS・ビルドなし」自体は**規模が問題なだけで思想としては悪くない**。ビルド不要・依存ゼロ・どこでも動く、は個人ツールの美徳。フレームワーク導入は目的ではなく、上記1-2を解く手段として評価する

---

## 2. 理想アーキテクチャ（目標像）

規模と用途に対する「ちょうど良い理想」は次の4層。

```
┌────────────────────────────────────────────┐
│ UI 層: 画面ごとのコンポーネント（Svelte 5）      │  ← 現 ui.js/report.js/schedule.js
│   ・インラインハンドラ廃止・window露出ゼロ      │
├────────────────────────────────────────────┤
│ アプリ状態層: ストア（$state / 現 state.js 相当） │  ← mutator API に一本化（P2-1）
│   ・undo/redo（history）はここに寄生           │
├────────────────────────────────────────────┤
│ ドメイン層: 純粋関数のビジネスロジック           │  ← detectDiff / calculateProgress /
│   ・TypeScript・DOM非依存・単体テストの主戦場   │     getWorkingDays / 丸め・日付
├────────────────────────────────────────────┤
│ 永続化層: Repository インターフェース           │  ← 現 storage.js を抽象化
│   実装1: IndexedDB (Dexie)  ← まずこれ        │
│   実装2: 同期API (Mac mini + Tailscale) ←任意  │
└────────────────────────────────────────────┘
```

設計原則:
- **ドメイン層を最優先で純化する**。工数計算・差分検出・営業日計算は完全にDOM/ストレージ非依存の TypeScript 関数にする。ここが太いほどテストが効き、UIフレームワークを将来何に替えても資産が残る
- **永続化は Repository パターンで抽象化**。「localStorage → IndexedDB → 同期API」をUI無改修で差し替えられる形に。スキーマにバージョン番号を持たせ、マイグレーション関数を必須にする（現データからの移行もこの仕組みの第1号として書く）
- **UIは画面単位のコンポーネント**。タブ1枚＝1コンポーネント群。全件 innerHTML 置換はフレームワークの差分描画に自然に置き換わる（P3-6 の難所が消える）

## 3. 技術スタック推奨

### 推奨構成（理想形）

| 領域 | 推奨 | 根拠 |
|---|---|---|
| 言語 | **TypeScript**（段階導入: まず `checkJs` + JSDoc → 新規から .ts） | LLM協働開発での誤り検出が桁違い。24k行の暗黙スキーマ（estimates/actuals の形）を型として明文化できる |
| ビルド | **Vite** | 事実上の標準。設定最小・GH Pages 出力も一発。`lib/` 同梱運用から npm 管理へ |
| UIフレームワーク | **Svelte 5** | (a) ランタイム最小・出力が素のJSに近い、(b) `$state` ルーンが現 state.js の発想と地続き、(c) HTMLテンプレートが現資産の移植先として最も距離が近い。**代替: Preact**（React知識を優先するなら）／**Lit**（脱フレームワーク思想を貫くなら） |
| 状態管理 | Svelte ルーン（追加ライブラリなし） | この規模で Redux 類は過剰 |
| 永続化 | **IndexedDB + Dexie** | 容量制約解消・非同期・インデックス。localStorage からの移行関数を初回起動で実行 |
| テスト | **Vitest**（ドメイン層）＋ **Playwright**（主要3フローのスモーク: 見積登録→実績入力→レポート表示） | 現 node:test 48件は Vitest へほぼ機械移植可能 |
| Lint/整形 | ESLint（既設を強化）+ Prettier | P1-5 と同じ |
| Excel/祝日 | 現行踏襲（xlsx.mjs / japanese-holidays を npm 経由に） | 実績があり替える理由がない |
| LLM分析 | 現行踏襲（Ollama）。接続先を `localhost` 固定から設定値へ | 後述のデプロイ案Bで iPhone からも使えるようになる |

### 選ばない選択肢と理由（重要）

- **React + Next.js 等のフルスタックFW**: SSR・ルーティング・サーバーは不要。個人ローカルツールには運用負債が増えるだけ
- **クラウドDB（Supabase/Firebase等）**: 無料枠は「時間で変わる情報」であり、課金ゼロ主義と相性が悪い（news での教訓）。個人の就労データを外部に置くプライバシー上の理由もある
- **Electron/Tauri 化**: 配布の必要がなく、ブラウザ＋ホーム画面追加で足りている
- **ゼロベース書き直し（ビッグバン）**: 動いている24k行・機能多数・仕様書なし。書き直しは必ず仕様の取りこぼしを生む。**強く非推奨**（§5の段階移行で同じ理想形に到達できる）

## 4. デプロイ・配信の選択肢

| 案 | 構成 | プライバシー | コスト | 備考 |
|---|---|---|---|---|
| A: 現状維持 | GitHub Pages（PUBLICリポ） | アプリ公開・データ端末内 | 0 | ブランチプレビューが便利。**コードが公開である自覚だけ持つ** |
| **B: 推奨** | **Mac mini + Tailscale 配信**（news と同居運用） | **完全私有**（tailnet限定） | 0 | news の static-server/launchd 資産を流用可。**Ollama に iPhone からも到達可**（`OLLAMA_HOST=0.0.0.0` 化＋アプリの接続先設定化が必要）。CI は GitHub Actions のまま、デプロイは Mac mini が `git pull`（launchd 定期 or webhook） |
| C: Cloudflare Pages | プライベートリポでも無料でPages相当 | アプリ非公開化は可・データ端末内 | 0 | GitHub Pages の「PUBLIC必須」制約だけを解きたい場合の中間解 |

**推奨は B**。理由: (1) 工数データを扱うツールの配信をtailnetに閉じられる、(2) news で実証済みの運用（launchd・静的配信・iPhoneホーム画面）をそのまま使える、(3) LLM分析がiPhoneからも動くようになり機能が増える、(4) 将来「同期API」（§2 実装2）を置く場所も同じMac miniになる＝アーキテクチャの終着点と配信基盤が一致する。
移行中は A(main=安定版の公開プレビュー) と B(私有本番) の併存も可能。

### 同期API（任意・最終フェーズ）

複数端末で同一データを使いたくなった時点で、Mac mini に極小の同期層を足す:
- **形**: SQLite + 数エンドポイントのAPI（Hono/Node か FastAPI/Python・news の explain_gateway と同パターンの launchd 常駐）
- **同期方式**: 端末→サーバの単方向バックアップから始め、必要になったら Last-Write-Wins の双方向へ。CRDT(Yjs等)は単独利用者には過剰
- **これも課金ゼロ**（tailnet内で完結・クラウド不使用）

## 5. 移行ロードマップ（ストラングラー方式・各段階で価値が完結）

「大規模リファクタリング」を一撃でやらず、**常に動く状態を保ったまま太い幹を移し替える**。各フェーズは単独で打ち切っても損しない順に並べてある。

| Phase | 内容 | 得られる価値 | 前提 |
|---|---|---|---|
| 0 | refactoring-proposals の P1（重複統合・デッドコード・日付共通化・ESLint強化） | 足場の整地 | 済んでいる特性テスト48件 |
| 1 | **Vite + Vitest + TypeScript(checkJs) 導入**。既存JSは無改修でビルド対象化、`lib/` を npm 依存へ | 型検査が全ファイルに掛かる・バンドラが未参照コードを可視化・テスト環境近代化 | なし（挙動不変） |
| 2 | **ドメイン層の抽出**: 工数計算・差分・営業日・丸めを `src/domain/*.ts` へ移動（re-exportファサードで挙動不変）＋テスト拡充（proposals の横断ロードマップ） | 資産の恒久化。以後どのUI技術でも使い回せる | Phase 1 |
| 3 | **永続化の Repository 化 + IndexedDB 移行**: storage.js を interface 化 → Dexie 実装 → 初回起動時に localStorage から自動移行（旧データはバックアップキーとして温存） | 容量制約・スキーマ移行問題の解消 | Phase 2（スキーマの型定義） |
| 4 | **UIの画面単位 Svelte 化**: 依存の少ない画面（設定・休暇管理あたり）から1画面ずつ。移行済み画面はインラインハンドラ・window露出を撤去 | ui.js 4,275行の段階的解体。P2-3/P2-4 がフレームワークの標準形で解決 | Phase 1〜3 |
| 5 | **デプロイB切替**（Tailscale私有配信）＋ Ollama 接続先の設定化 | 完全私有・iPhoneでLLM分析可 | いつでも可（Phase番号と独立） |
| 6 | （必要になったら）同期API | マルチデバイス | Phase 3 |

工数感（Claude Code 協働前提の目安）: Phase 1 = 半日〜1日、Phase 2-3 = 各数日、Phase 4 = 画面ごとに0.5〜2日×十数画面（数週間・分割可能）、Phase 5 = 半日。

## 6. 判断を仰ぎたい分岐点（実施前に決めること）

1. **マルチデバイス同期は要るか** — 要らないなら Phase 3 で IndexedDB 止まり・Phase 6 不要。要るなら Repository 設計時に同期前提の形（更新タイムスタンプ・tombstone）を仕込む
2. **配信の私有化（案B）を優先するか** — コード変更ゼロで今すぐでも移行可能。優先度だけの問題
3. **UIフレームワークの好み** — 推奨は Svelte 5 だが、React の知識資産を優先するなら Preact でロードマップは同一
4. **Phase 4 をどこまでやるか** — 全画面移行が理想だが、「新機能と修正が多い画面だけ移行し、安定画面は現行のまま」という止め方も健全

## 7. まとめ（1行版）

**書き直さずに、ドメイン層→永続化→UIの順で幹を移し替え、最終形は「TypeScript + Svelte 5 + Vite + IndexedDB、Mac mini/Tailscale 私有配信、必要になったら同一Mac miniに同期API」— 全工程課金ゼロ。**
