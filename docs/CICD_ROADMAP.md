# CI/CD ロードマップ

GitHub Actions による自動チェック（CI）と自動公開（CD）の段階的な導入計画。

このプロジェクトは **ビルド工程なし・依存パッケージなし（素の HTML/CSS/JS）** が持ち味のため、
重いツールを最初から全部入れるのではなく、**効果の高いものから段階的に**積む方針とする。

| 段階 | 状態 | 概要 |
|------|------|------|
| レベル1 | ✅ 導入済み | 公開前の最小チェック（依存ゼロ） |
| レベル2 | 🔄 一部導入 | ESLint(no-undef) を導入済み。Prettier 等の残りは案 |
| レベル3 | 📋 案 | E2E・セキュリティ・CD の作り直し |

---

## 前提：現状の CD（公開の仕組み）

- `.github/workflows/deploy.yml` が `main` への push（または手動）で GitHub Pages へ公開する。
- `main` の内容をルートに、各実験ブランチ（sandbox / redesign / ui-scaling）を `preview/<名前>/` 配下に展開する。
- 実験ブランチの変更は「`main` へ空コミットを push して発火」させる運用。

つまり **CD は元からあるが、公開前に内容を検証する CI が無かった**。レベル1 はその穴を埋めるもの。

---

## レベル1（導入済み）— 公開前の最小チェック

ファイル: `.github/workflows/ci.yml`

PR を開いた時と `main` への push 時に走る。**依存パッケージを一切増やさず**、Node 標準機能だけで動く。

| チェック | 内容 | 防げる事故 |
|----------|------|-----------|
| JS 構文チェック | `js/**/*.js` を `node --check` にかける（ESM 判定のため一時 `.mjs` 化） | 構文ミスでページ全体が真っ白になる |
| 参照ファイル実在チェック | `index.html` がローカル参照する `script`/`link` の実ファイル有無を確認 | リネーム漏れによる読み込み失敗（白画面） |

### 設計上の判断

- **対象は `js/` のみ**。`lib/`（外部同梱ライブラリ）はチェック対象外。自作コードに絞る。
- ESM を CommonJS と誤判定させないため、各ファイルを一時的に `.mjs` にコピーして `node --check` する。
- `package.json` を作らない（レベル1 では依存ゼロを維持）。

### 今後この CI を有効に保つための注意

- 新しい JS ファイルは `js/` 配下に置けば自動的にチェック対象になる。
- `index.html` に新しいローカル参照を足したら、ファイルが実在することを確認する（CI でも検出される）。

## レベル2（一部導入）— 書き方の統一と既存検証の自動化

「同じケアレスミスを繰り返す」「レビューで体裁の指摘が増える」と感じ始めたら検討する。

ここで初めて `package.json`（開発用依存のみ）を 1 つ導入する。これは「依存ゼロ」方針を初めて崩す変更だが、
**ESLint(no-undef) は白紙化事故を機に前倒しで導入済み**（経緯は後述）。残りの項目（Prettier 等）は引き続き案。

| 追加項目 | 内容 | コスト |
|----------|------|--------|
| ESLint | ✅ **導入済み**（`ci.yml` の `lint` ジョブ / `no-undef` 中心） | 設定ファイル + 依存追加（eslint・globals）。既存の window 経由グローバルは globals 登録で対応 |
| Prettier | コード整形の統一 | 既存コードに一括適用すると差分が大きくなる点に注意 |
| 既存検証スクリプトの実行 | `llm-analysis/tests/verify_js_summarize.mjs`（JS版とPython版の出力一致を検証）を CI で実行 | Python3 のセットアップが必要（`actions/setup-python`） |
| PR プレビュー公開 | PR ごとに専用 URL を発行し、レビュー時に実物を確認 | デプロイ先・URL 設計の検討が必要 |

### メモ
- ESLint/Prettier は「フラットコンフィグ（`eslint.config.js`）」で最小構成から始めるのが無難。
- 既存コードへの一括整形は別 PR に分け、機能変更の差分と混ぜない。

### ESLint 導入の経緯（2026-06-16）

`nonProjectWork` 未定義参照で `loadData()` が ReferenceError を起こし、実績一覧が**全画面白紙**になる事故が発生（`/preview/ui-scaling/`）。

- レベル1 の `node --check`（構文チェック）は「構文は正しいが未定義参照」型を検出できず素通りしていた。
- ESLint `no-undef` なら静的に検出できることを実証できたため、レベル2 のうち **ESLint のみを前倒し導入**した。

**導入内容**:
- `package.json`（devDependencies は `eslint` / `globals` のみ）＋ `package-lock.json`
- `eslint.config.mjs`: フラットコンフィグ。`globals.browser` ＋ 既存の window 経由グローバル（`phaseCollapsed`・`JapaneseHolidays`）をホワイトリスト登録。ルールは `no-undef` 中心（整形系は入れない）
- `ci.yml` に `lint` ジョブを追加（既存 `checks` の依存ゼロは温存し、`npm` は `lint` ジョブだけに閉じる）。`push` 発火対象に `experiment/ui-scaling` を追加（事故も ui-scaling 直 push だったため）

**運用上の注意**:
- import せず window 経由で bare 参照する新しいグローバルを足したら、`eslint.config.mjs` の `appGlobals` / `libGlobals` に登録する（さもないと `no-undef` で赤くなる）。
- 既存の bare 参照（`phaseCollapsed` 等）は本来 import に直すのが望ましく、将来の整理対象。

## レベル3（案）— E2E・セキュリティ・CD 作り直し

「タブが開かない等の事故が増えた」「依存ライブラリの脆弱性を継続的に見たい」と感じたら検討する。
作るのも維持するのも手間が大きいため、**規模に対して過剰にならないか**を都度判断する。

| 追加項目 | 内容 | コスト |
|----------|------|--------|
| Playwright スモークテスト | ページが開き、主要タブ（見積・実績・レポート・スケジュール等）が表示されるかを自動確認 | テストの作成・維持コスト大。ブラウザバイナリの取得時間も |
| CodeQL | コードの脆弱性スキャン（`SECURITY_AUDIT_REPORT.md` の問題意識と合致） | 純粋な静的サイトでは検出対象が限られる場合あり |
| Dependabot | 同梱ライブラリ更新の追従（手動バンドル運用との相性は要検討） | CDN 不使用・手動同梱のため、運用設計が必要 |
| CD リファクタ | 「`main` へ空コミットして発火」運用の見直し、ブランチ別ワークフロー化 | 既存 `deploy.yml` の挙動を壊さないよう慎重に |

---

## 採用判断の目安（まとめ）

- **レベル1**: 常設（導入済み）。
- **レベル2**: 体裁の指摘や同種ミスが増えたら。`package.json` 導入の是非とセット。
- **レベル3**: 表示崩れ事故の増加、または継続的なセキュリティ監視が必要になったら。

最初から全部入れる必要はない。**実際に困りごとが出てから、その困りごとに対応する段を足す**。
