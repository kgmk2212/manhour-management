## なぜやるか

実績入力時のエラー（誤った時間、重複、未来日）を保存前に検出してユーザーに確認することで、
後から「データを確認したら数値が狂っていた」という事態を防ぐ。

H-24-1と H-24-3 を統合し、共通の `validation.js` として基盤化することで、
今後追加される入力系機能にも同じ検証ロジックを適用できるようにする。

## 設計判断

### 採用する方針
設計書の閾値をベースに定数化:

| 項目 | エラー閾値 | 警告閾値 |
|------|----------|---------|
| 1件の時間 | > 24h | > 12h |
| 1日の合計 | > 24h | > 10h |
| 最小時間 | < 0.1h | - |
| 未来日 | - | 今日より後 |
| 重複 | - | 同日・同タスク・同工程・同メンバー |

### 設計書から変更する点
| 項目 | 設計書 | 実装 | 理由 |
|------|--------|------|------|
| 重複時の動作 | 専用の3択ダイアログ | 標準confirm（「このまま登録しますか？」） | まず最小実装。後にカスタムダイアログで強化可能 |
| 複数警告の表示 | 個別ダイアログ | 1つの確認ダイアログに集約 | ダイアログ連鎖は鬱陶しい |
| エラー/警告の分離 | 設計書では曖昧 | errors（保存阻止）/ warnings（確認）で明確化 | 後から拡張しやすい |

## データフロー

```
ユーザー送信
   ↓
existing guard (required fields check)
   ↓
validateActualInput(actual, { excludeId })
   ↓
  返す:
   - errors:     [ '時間を入力してください', ... ]
   - warnings:   [ '12時間を超えています', ... ]
   - duplicates: [ { id, date, ... } ]
   - isValid: errors.length === 0
   ↓
confirmValidationResult(result)
   ↓
  - errors あり → alert 表示 → false（中止）
  - warnings のみ → confirm で確認 → ユーザー選択
  - なし → true（そのまま保存）
   ↓
save 処理
```

## 動作確認項目

### エラー（保存阻止）
- [ ] 時間を0にして保存 → アラート「0.1h以上で入力してください」
- [ ] 時間を25にして保存 → アラート「24時間を超える入力はできません」
- [ ] 同日合計が24hを超える入力 → アラート

### 警告（確認ダイアログ）
- [ ] 1件で15時間入力 → 「長めです」確認
- [ ] 未来日（明日）で入力 → 「未来の日付です」確認
- [ ] 同日合計が11hになる入力 → 「合計が11.0hになります」確認
- [ ] 同日・同タスク・同工程・同メンバーで重複 → 「既にN件登録されています」確認

### 複合ケース
- [ ] 未来日 + 長時間 → 複数の警告が1つのダイアログにまとまる
- [ ] 編集時に自身を重複として検出しない
- [ ] キャンセルを押すとデータが追加されない

### 既存動作への影響
- [ ] 通常の入力（8h、今日、初回）はダイアログなしで保存される
- [ ] 必須項目の既存チェックは引き続き動作する
- [ ] undo/redoが正常に動作する
