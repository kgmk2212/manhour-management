## 1. テンプレート機能

### 現状の課題
- 毎日似た作業をするメンバーが、同じバージョン/タスク/工程/時間を毎回手入力している
- 「定例会議2h」「コードレビュー1h」のような定型作業の入力が特に冗長
- 入力の手間がデータ精度低下（入力サボり）の原因になっている

### 提案内容
- 頻出する入力パターンをテンプレートとして保存・呼び出しできる機能
- テンプレートの一括適用（「今日はこのテンプレートセット」でワンタップ入力完了）

### 実装方針

#### データモデル
```javascript
// js/state.js に追加
// localStorage キー: manhour_templates
const templates = [
  {
    id: "tmpl_001",
    name: "定例会議",
    icon: "meeting",           // オプション: 視覚的識別子
    entries: [
      {
        version: "v2.0",
        task: "管理",
        process: "その他",
        member: "森",
        hours: 2.0
      }
    ],
    isSet: false,              // true: 複数エントリのセット
    frequency: "daily",        // daily / weekly / manual
    weekdays: [1, 3, 5],      // frequency=weekly時の曜日（月水金）
    usageCount: 42,            // 利用回数（ソート用）
    lastUsed: "2026-04-13",
    createdAt: "2026-03-01"
  }
];
```

#### UI変更

**クイック入力タブに追加**:
```
┌─────────────────────────────────────────────┐
│ [実績入力] [見積登録] [休暇登録] [テンプレ]  │  ← 4つ目のセグメント追加
├─────────────────────────────────────────────┤
│                                             │
│  📋 よく使うテンプレート                      │
│  ┌──────────────────────┐                   │
│  │ 🔄 定例会議 (2h)     │  [適用]           │
│  │ 💻 開発作業 (8h)     │  [適用]           │
│  │ 📝 コードレビュー(1h)│  [適用]           │
│  └──────────────────────┘                   │
│                                             │
│  ⊕ 現在の入力内容をテンプレート保存           │
│  ⊕ 新規テンプレート作成                      │
│                                             │
│  📦 テンプレートセット                        │
│  ┌──────────────────────┐                   │
│  │ 通常の1日セット       │  [一括適用]       │
│  │  - 定例会議 2h        │                   │
│  │  - 開発作業 5h        │                   │
│  │  - レビュー 1h        │                   │
│  └──────────────────────┘                   │
└─────────────────────────────────────────────┘
```

#### 影響ファイル
| ファイル | 変更内容 |
|---------|---------|
| `js/state.js` | templates配列とgetter/setter追加 |
| `js/storage.js` | テンプレートの保存・読込処理 |
| `js/constants.js` | STORAGE_KEYS.TEMPLATES追加 |
| `js/ui.js` | テンプレートタブのレンダリング |
| `js/actual.js` | テンプレートからの実績作成処理 |

#### 実装ステップ
1. データモデル定義（state.js, storage.js, constants.js）
2. テンプレートCRUD処理の実装
3. テンプレートタブUIの構築
4. 「現在の入力をテンプレ保存」ボタンの実装
5. テンプレート適用処理（日付を今日に差し替えて実績作成）
6. テンプレートセット（複数エントリ一括適用）の実装
7. 利用頻度による自動ソート

### 技術的考慮事項

- バックアップ/リストアにテンプレートデータを含める

## 2. 前日コピー機能

### 現状の課題
- 多くの日は前日と似た作業をしているが、すべて最初から入力し直す必要がある
- 週末明けにまとめ入力する際、金曜の内容をベースに微調整したいケースが多い

### 提案内容
- 指定日の実績をコピーして新しい日付で入力する機能
- 時間のみ変更して素早く確定できるインライン編集

### 実装方針

#### UI変更
```
クイック入力 > 実績入力
┌────────────────────────────────────┐
│ 📅 2026-04-14 (月)                 │
│                                    │
│ [前日の実績をコピー] [他の日からコピー]│
│                                    │
│ コピー元: 2026-04-11 (金) の実績:   │
│ ┌────────────────────────────────┐ │
│ │ ☑ v2.0/設計/UI  森  3.0h [__]│ │ ← 時間を直接編集可能
│ │ ☑ v2.0/実装/PG  森  4.0h [__]│ │
│ │ ☐ v2.0/会議/他  森  1.0h [__]│ │ ← チェックで選択/除外
│ └────────────────────────────────┘ │
│                                    │
│ [選択した実績をコピー作成]            │
└────────────────────────────────────┘
```

#### 影響ファイル
| ファイル | 変更内容 |
|---------|---------|
| `js/actual.js` | copyActualsFromDate()関数の追加 |
| `js/ui.js` | コピーUIのレンダリング、日付ピッカー |
| `index.html` | コピーセクションのHTML |
| `style.css` | コピー確認リストのスタイル |

#### 実装ステップ
1. 指定日の実績取得・フィルタリング処理
2. コピー確認UIの構築（チェックボックス+時間編集）
3. コピー実行処理（新しい日付とIDで実績作成）
4. 「前日」ボタンの実装（直前の営業日を自動取得）
5. 「他の日から」ボタンの実装（日付ピッカー表示）

### 技術的考慮事項
- 祝日・休日をスキップした「前営業日」の計算（japanese-holidays.js活用）
- コピー元の実績が変更/削除されていた場合のハンドリング

### 工数見積
- 小〜中（1-2日）

## 3. 最近使った組み合わせの候補表示

### 現状の課題
- バージョン → タスク → 工程 → メンバーの順に毎回選択する手間
- 同じ組み合わせを日々繰り返し選択している
- ドロップダウンの選択肢が多いと目的のものを見つけにくい

### 提案内容
- 過去の入力履歴から「最近使った組み合わせ」をサジェスト表示
- ワンクリックでフォームに自動入力

### 実装方針

#### データモデル
```javascript
// localStorage キー: manhour_recent_combos
// 最大20件を保持、LRU（Least Recently Used）で古いものを削除
const recentCombinations = [
  {
    version: "v2.0",
    task: "ログイン画面",
    process: "PG",
    member: "森",
    lastUsed: "2026-04-14T10:30:00",
    count: 15
  }
];
```

#### UI変更
```
実績入力フォームの上部に追加:
┌────────────────────────────────────────┐
│ ⚡ 最近の入力                           │
│ ┌────────┐ ┌────────┐ ┌────────┐     │
│ │v2.0    │ │v2.0    │ │v1.5    │     │
│ │ログイン │ │API設計  │ │バグ修正 │     │
│ │PG/森   │ │UI/森   │ │PT/田中  │     │
│ └────────┘ └────────┘ └────────┘     │
│              ← 横スクロール →           │
└────────────────────────────────────────┘
```

#### 影響ファイル
| ファイル | 変更内容 |
|---------|---------|
| `js/state.js` | recentCombinations配列追加 |
| `js/storage.js` | 最近の組み合わせの永続化 |
| `js/actual.js` | 実績作成時に組み合わせを記録 |
| `js/ui.js` | サジェストチップのレンダリング |
| `style.css` | チップカード・横スクロールのスタイル |

#### 実装ステップ
1. 組み合わせ記録の仕組み（実績作成のフック）
2. LRU管理ロジック（20件超で古いものを削除）
3. チップUIのレンダリング（横スクロールコンテナ）
4. チップクリック → フォーム自動入力の実装
5. フォームのバージョン/タスク変更時にチップをハイライト

## 4. 見積 vs 実績の乖離アラート

### 現状の課題
- 見積を超過していることに気づくのがレポート画面を見た時で、すでに手遅れ
- 日々の入力時に「あとどれだけ余裕があるか」が見えない
- レポートの分析機能は充実しているが、リアルタイム性に欠ける

### 提案内容
- 実績入力時に「この見積の消化率」をリアルタイム表示
- 閾値（80%, 100%）到達時にトースト通知
- クイック入力画面に「危険なタスク」のサマリー表示

### 実装方針

#### UI変更
```
実績入力フォーム内:
┌──────────────────────────────────────────┐
│ バージョン: [v2.0 ▼]                      │
│ タスク:     [ログイン画面 ▼]               │
│ 工程:       [PG ▼]                        │
│                                           │
│ ┌─ 見積状況 ─────────────────────────────┐│
│ │ 見積: 40h  実績: 35h  残: 5h           ││
│ │ ████████████████████░░░░ 87.5%         ││
│ │ ⚠ 見積の80%を超過しています             ││
│ └────────────────────────────────────────┘│
│                                           │
│ 時間: [___] h                             │
│ ※ この入力で 95.0% (38h/40h) になります    │
└──────────────────────────────────────────┘
```

**ダッシュボード（クイック入力上部）**:
```
┌─ ⚠ 注意が必要なタスク ─────────────────┐
│ 🔴 v2.0/ログイン/PG  超過 +5h (112%)   │
│ 🟡 v2.0/API設計/UI   残り2h (93%)      │
│ 🟢 v1.5/テスト/PT    残り20h (60%)     │
└─────────────────────────────────────────┘
```

#### 影響ファイル
| ファイル | 変更内容 |
|---------|---------|
| `js/actual.js` | 入力時の消化率計算・表示 |
| `js/estimate.js` | 見積残取得APIの整備 |
| `js/ui.js` | 乖離アラートUIのレンダリング |
| `js/report.js` | 既存の乖離計算ロジックの共通化 |
| `js/constants.js` | 閾値定数（WARNING_THRESHOLD: 0.8, DANGER_THRESHOLD: 1.0） |
| `style.css` | プログレスバー・アラートバッジのスタイル |

#### 実装ステップ
1. 見積消化率計算の共通関数を作成（report.jsの既存ロジックを抽出）
2. 実績入力フォームにプログレスバーを追加
3. バージョン/タスク/工程の選択変更時に消化率を動的更新
4. 「この入力後の予測値」をリアルタイム計算
5. 閾値超過時のトースト通知
6. ダッシュボードの「注意タスク」サマリーウィジェット

### 技術的考慮事項
- report.jsの既存ロジックとの重複を避ける（共通関数に抽出）
- パフォーマンス：選択変更のたびに全データスキャンしないようインデックス利用
- 閾値のカスタマイズ（設定画面で変更可能に）

### 工数見積
- 中（2-3日）

## 5. バッチ入力モード

### 現状の課題
- 毎日入力する習慣がないメンバーが、週末にまとめ入力したい
- 1日ずつ日付を切り替えて入力するのは面倒
- 「月曜〜金曜、全部同じ作業だった」ケースの入力が5回分の手間

### 提案内容
- 週間ビューで月〜金の実績を一覧表示・一括編集
- 行（タスク）× 列（曜日）のグリッドで時間を直接入力

│ v2.0/設計/UI   │ 3.0  │ 2.0  │ [__] │ [__] │ [__] │ 5.0 │

## 6. 自動保存

### 現状の課題
- 明示的な「保存」操作を忘れるとデータが失われるリスク
- ブラウザクラッシュや誤ってタブを閉じた場合のデータロス

### 提案内容
- フォーム入力中のデータをリアルタイムでlocalStorageに一時保存
- 一定時間操作がなかった場合に自動確定保存
- 保存状態のインジケーター表示

### 実装方針

#### データモデル
```javascript
// localStorage キー: manhour_draft
const draft = {
  type: "actual",              // actual / estimate / vacation
  data: { /* フォーム入力中のデータ */ },
  savedAt: "2026-04-14T10:30:00",
  autoSaveEnabled: true
};
```

#### UI変更
```
ヘッダー右上:
┌──────────────────────┐
│ ✓ 保存済み 10:30      │  ← 通常時
│ ● 保存中...           │  ← 保存中
│ ⟳ 未保存の変更あり     │  ← 変更検出時
└──────────────────────┘
```

#### 実装ステップ
1. フォーム変更検出（input/changeイベントのデバウンス監視）
2. ドラフトデータの一時保存（debounce 2秒）
3. ページ読込時にドラフト復元の確認ダイアログ
4. 保存状態インジケーターの実装
5. 設定画面で自動保存のON/OFF切替

### 工数見積
- 小〜中（1-2日）

## 7. グローバル検索

### 現状の課題
- タスク名やメンバー名でデータを横断検索する手段がない
- 「あのタスクの見積はどこだっけ？」→ 画面を切り替えてフィルターして探す手間
- バージョン数・タスク数が増えると情報を見つけにくい

### 提案内容
- `Ctrl+K` / `Cmd+K` でグローバル検索パレットを表示
- タスク名・メンバー名・バージョン名で横断検索
- 検索結果から直接該当画面に遷移

### 実装方針

│   ログイン画面 (v2.0)          → 実績を見る  │

#### 実装ステップ

- 日本語のひらがな/カタカナ/漢字の横断検索
- 検索インデックスの更新タイミング（データ変更時に再構築）
- モバイルではヘッダーの検索アイコンから起動

## 9. ショートカットキー

### 現状の課題
- すべての操作がマウス/タッチ前提で、キーボードだけで完結できない
- パワーユーザーにとって操作速度のボトルネック

### 提案内容
- グローバルショートカット（タブ切替、新規入力、検索）

## 12. デフォルト値の学習

### 現状の課題
- フォームを開くたびに全フィールドが空の状態から始まる
- 「自分の名前」「よく使うバージョン」など、ほぼ毎回同じ値を選択

### 提案内容
- 直近の入力パターンから最も確率の高いデフォルト値を自動設定
- メンバー名は前回の値を記憶
- バージョンは最新のアクティブバージョンをデフォルトに

### 実装方針

#### データモデル
```javascript
// localStorage キー: manhour_user_defaults
const userDefaults = {
  member: "森",                        // 最後に使用したメンバー
  version: "v2.0",                     // 最後に使用したバージョン
  defaultHours: 1.0,                   // デフォルト時間
  frequentProcessByTask: {             // タスク別の頻出工程
    "ログイン画面": "PG",
    "API設計": "UI"
  }
};
```

#### 実装ステップ
1. ユーザーデフォルトのデータモデル定義
2. 実績/見積作成時にデフォルト値を更新
3. フォーム表示時にデフォルト値を適用
4. タスク選択時に頻出工程を自動選択
5. 設定画面でデフォルトのリセット

### 工数見積
- 小（1日）
