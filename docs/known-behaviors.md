# 既知の怪しい挙動（要確認・未修正）

2026-07-06 の特性テスト導入（`tests/`・48件）の過程で発見した「バグの可能性があるが、現状の振る舞いのままテストで固定してある」挙動の記録。**修正する場合は挙動変更になるため、対応するテストの期待値も同時に更新すること**（テストが落ちることで挙動変更に気づける、というのがこの固定の意図）。

すべて実機（Node v23 / 実モジュール import）で観測した実際の値。

---

## 1. `roundNum()` — カンマ区切り数値文字列が黙って 0 になる

- **場所**: `js/merge-core.js` の `roundNum()`
- **観測**: `roundNum("5,000")` → `0`（`Number("5,000")` が `NaN` になり 0 へフォールバック）
- **リスク**: Excel からの貼り付けや手入力で `1,５00` のような3桁区切りの工数が入った場合、**エラーにならず工数 0 として差分判定される**。「値が消えた」事故に直結しうる、3件の中では最も実害リスクが高い挙動
- **修正案**: `Number()` に渡す前にカンマ除去（`String(v).replace(/,/g, "")`）するか、`NaN` の場合はフォールバックせず呼び出し側へ明示的にエラー/警告を返す
- **固定テスト**: `tests/merge-core.test.js`（roundNum の項）

## 2. `roundNum()` — 浮動小数点による丸めの非一貫性

- **場所**: 同上（実装は `Math.round(v * 100) / 100`）
- **観測**:
  - `roundNum(3.005)` → `3.01` （切り上がる）
  - `roundNum(1.005)` → `1` （切り上がらない）
  - `roundNum(-1.005)` → `-1` （-1.01 にならない）
- **原因**: IEEE754 で `1.005 * 100 = 100.49999...` となる典型的な浮動小数点誤差。JS の `Math.round` ベースの2桁丸め全般に共通する既知の特性
- **リスク**: 工数は 0.25h/0.5h 刻みの運用が主のため実害はほぼ無い。ただし将来「0.005h 単位」を扱うと差分判定（`valueEq`）が境界値で不安定になる
- **修正案**: 対応するなら `Math.round((v + Number.EPSILON) * 100) / 100`、または文字列経由の丸め。**急ぎ対応は不要**
- **固定テスト**: `tests/merge-core.test.js`

## 3. `parsePartialJson()` — 末尾の不完全エスケープシーケンスが欠落する

- **場所**: `js/partial-json.js` の `parsePartialJson()`
- **観測**:
  - `{"a":"esc\` （文字列値の末尾がエスケープ開始 `\` で切断） → `{a: "esc"}` （`\` ごと削除）
  - `{"a":"x\u00` （Unicode エスケープ途中で切断） → `{a: "x"}` （シーケンスごと削除）
- **リスク**: 用途が「Ollama ストリーミング応答の途中表示」なので、次のチャンクが来れば完全な JSON で再パースされる＝**表示上の一瞬の欠落のみで実害なし**。最終結果の保存には完全 JSON を使っている限り問題にならない
- **注意**: もし `parsePartialJson` の結果を「最終値」として保存する経路を将来作る場合は、この末尾欠落が永続化されるため要対策
- **固定テスト**: `tests/partial-json.test.js`
