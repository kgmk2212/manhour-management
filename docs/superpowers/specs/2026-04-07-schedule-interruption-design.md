# スケジュール中断・差し込み機能 設計書

> **作成日**: 2026-04-07
> **ブランチ**: experiment/redesign
> **フェーズ**: Phase 1（モーダル＋コンテキストメニュー）、Phase 2（D&D差し込み）

---

## 1. 概要

### 背景

ガントチャートのスケジュール管理において、作業を中断して別の作業を差し込む必要がある場面は2種類ある:

- **割り込み型**: 作業中に緊急対応が入り、途中で中断→差し込み→再開
- **計画変更型**: 事前に「この期間は別作業を入れたい」と計画的に分割

### 目的

ガントチャートのバーを分割し、差し込み作業の工数分だけ後続を自動的にずらす機能を実装する。

### スコープ

| フェーズ | 内容 | 状態 |
|---------|------|------|
| Phase 1 | コアロジック＋モーダル＋コンテキストメニュー | 今回実装 |
| Phase 2 | ドラッグ&ドロップによる差し込み | 後日実装 |

Phase 1だけで割り込み・計画変更の両方に対応できる。Phase 2はPhase 1の動作検証後に追加する。

## 2. データモデル

### 2.1 スケジュールオブジェクトの拡張

既存のスケジュールオブジェクトに `interruptions` フィールドを追加する。

```javascript
{
  // --- 既存フィールド（変更なし） ---
  id: "sch_001",
  version: "1.0",
  task: "機能A",
  process: "PG",
  member: "田中",
  startDate: "2026-04-07",
  endDate: "2026-04-18",       // ← 中断を考慮して再計算される
  estimatedHours: 40,
  status: "pending",
  color: "#4A90D9",
  note: "",

  // --- 新規フィールド ---
  interruptions: [
    {
      id: "int_1712345678",           // 中断ID（int_ + timestamp）
      splitDate: "2026-04-09",        // 中断した日（この日の作業終了後に中断）
      consumedHours: 24,              // 中断時点の消化工数（自動計算、手動調整可）
      reason: "緊急修正対応",          // 中断理由
      insertedScheduleId: "sch_002"   // 差し込みスケジュールのID（null可）
    }
  ]
}
```

### 2.2 設計判断

- `interruptions` は配列。複数回の中断に対応する
- `consumedHours` は `startDate` 〜 `splitDate` の営業日数 × `hoursPerDay` で自動計算し、手動調整も可能
- `insertedScheduleId` は任意（null可）。理由だけ記録して差し込みスケジュールは後で作成することもできる
- `endDate` は中断を考慮して再計算される（中断期間＋差し込み期間分だけ後ろにずれる）
- 差し込みスケジュールは通常のスケジュールオブジェクトとして作成される（特殊な型ではない）

### 2.3 endDate再計算ルール

```
元のendDate = calculateEndDate(startDate, estimatedHours, member)

中断がある場合:
  各セグメントの営業日を積み上げて計算

  例: 40h（5営業日）、中断2日の場合
  セグメント1: 4/7-4/9 (3日, 24h)
  中断期間:    4/10-4/11 (差し込み作業)
  セグメント2: 4/14-4/15 (2日, 16h) ← 残り
  → endDate = 4/15
```

### 2.4 セグメント計算（複数中断対応）

```javascript
// calculateSegments(schedule) の出力
// 中断が2回ある場合:
interruptions: [
  { splitDate: "4/9",  consumedHours: 24, insertedScheduleId: "sch_002" },
  { splitDate: "4/17", consumedHours: 32, insertedScheduleId: "sch_003" }
]

// → セグメント:
[
  { startDate: "4/7",  endDate: "4/9",  hours: 24, index: 0 },  // 0〜24h
  { startDate: "4/14", endDate: "4/15", hours: 8,  index: 1 },  // 24h〜32h
  { startDate: "4/21", endDate: "4/22", hours: 8,  index: 2 }   // 32h〜40h
]
```

各セグメントの工数は `consumedHours` の差分で算出:
- seg1: `interruptions[0].consumedHours` = 24h
- seg2: `interruptions[1].consumedHours - interruptions[0].consumedHours` = 8h
- seg3: `estimatedHours - interruptions[1].consumedHours` = 8h

### 2.5 セグメント開始日の決定

各セグメントの `startDate` は以下のルールで決定する:

- **seg1**: スケジュールの `startDate` をそのまま使用
- **seg2以降**: 前の中断に `insertedScheduleId` がある場合
  → 差し込みスケジュールの `endDate` の翌営業日
- **seg2以降**: 前の中断に `insertedScheduleId` がない場合（理由のみ記録）
  → 前のセグメントの `endDate`（= `splitDate`）の翌営業日

### 2.6 consumedHoursとsplitDateの関係

`splitDate` はカレンダー上の中断日を記録するメタデータ。`consumedHours` がセグメント計算の基準値。

ユーザーが `consumedHours` を手動調整した場合（例: splitDate=4/9だが consumedHours=20h に変更）、セグメントの工数は `consumedHours` に従う。`splitDate` は中断の日付記録・表示目的で保持する。

## 3. コアロジック

### 3.1 中断追加: `addInterruption(scheduleId, splitDate, reason, insertOptions?)`

```
処理フロー:
  1. 消化工数の自動計算
     startDate〜splitDate の営業日数 × hoursPerDay
     例: 4/7〜4/9 = 3日 × 8h = 24h

  2. 中断レコード作成
     { id, splitDate, consumedHours: 24, reason, insertedScheduleId: null }

  3. 差し込みスケジュール作成（insertOptions指定時）
     insertOptions: { version, task, process, hours }
     → 新スケジュール作成（startDate = splitDateの翌営業日）
     → insertedScheduleId に紐づけ

  4. endDate再計算
     残り工数 = estimatedHours - consumedHours
     差し込み終了日の翌営業日から残り工数分を配置
     → 新しいendDateを算出

  5. 連鎖ずらし（cascadeShift）
```

### 3.2 連鎖ずらし: `cascadeShift(changedSchedule)`

```
処理フロー:
  1. 対象の特定:
     a) 同じ version + task で、後続の工程（processOrder: UI→PG→PT→IT→ST）
     b) 同じ担当者の後続スケジュール（startDate >= 旧endDate）

  2. ずらし量の計算:
     diff = 新endDate - 旧endDate（営業日ベース）
     後続の startDate を diff 分後ろにずらし
     → endDate も再計算

  3. 再帰的に適用:
     ずらした各スケジュールに対しても cascadeShift を実行
     → 依存チェーン全体が整合性を保つ

  ※無限ループ防止: 処理済みIDをSetで管理
```

### 3.3 影響分析: `analyzeImpact(scheduleId, splitDate, insertHours)`

`cascadeShift` と同じロジックで影響範囲を計算するが、実際には変更せず影響リストを返す。確認ダイアログ用。

```javascript
// 戻り値の例:
[
  {
    schedule: { id: "sch_003", version: "1.0", task: "機能A", process: "PT" },
    member: "田中",
    oldStart: "2026-04-16", newStart: "2026-04-21",
    oldEnd: "2026-04-18", newEnd: "2026-04-23"
  },
  {
    schedule: { id: "sch_004", version: "1.0", task: "機能A", process: "IT" },
    member: "佐藤",
    oldStart: "2026-04-21", newStart: "2026-04-24",
    oldEnd: "2026-04-23", newEnd: "2026-04-25"
  }
]
```

### 3.4 中断取り消し: `removeInterruption(scheduleId, interruptionId)`

```
処理フロー:
  1. interruptions配列から該当中断を削除
  2. endDateを再計算（中断なしの元の長さに戻る）
  3. 差し込みスケジュールの扱い:
     → 差し込みスケジュール自体は削除しない
     → insertedScheduleId の紐づけだけ解除
     → 確認ダイアログ: 「差し込みスケジュールも削除しますか？」
  4. 逆方向の連鎖ずらし（前に詰める）
     → 同様に影響プレビュー → 確認 → 適用
```

## 4. UI/UX

### 4.1 分割バーの描画（Canvas）

`drawScheduleBar(schedule, rowY)` を拡張する。

```
描画フロー:
  interruptions が空 → 従来通り1本のバーを描画（変更なし）

  interruptions がある → セグメント分割描画:

  1. セグメント計算: calculateSegments(schedule)

  2. 各セグメントを独立したバーとして描画:
     ┌─────────┐             ┌─────────────┐
     │██ PG 24h│             │░░ PG 16h    │
     └─────────┘             └─────────────┘

  3. ✂マーク描画:
     セグメント1の右端に ✂（切断側）
     セグメント2の左端に ✂（再開側）

  4. 点線接続描画:
     セグメント1の右端 ─ ─ ─ セグメント2の左端
     色: バーと同色、opacity: 0.4
     線種: dashArray [4, 4]
```

### 4.2 クリック判定

分割バーの場合、各セグメントを個別の矩形として `scheduleRects` に登録する。クリック時にどのセグメントがクリックされたかも判別できるようにする。

### 4.3 操作方法（Phase 1）

#### コンテキストメニュー（右クリック or 長押し）

バー上での右クリックまたは長押しでメニューを表示:

```
┌─────────────────┐
│ 📋 詳細を表示    │
│ ✂ ここで中断     │  ← クリック位置の日付で分割
│ ✏️ 編集          │
│ 🗑 削除          │
└─────────────────┘
```

「ここで中断」を選択すると、クリック位置から `xToDate()` で日付を算出し、中断モーダルが開く。

#### スケジュール詳細モーダルから

既存のスケジュール詳細モーダルに中断履歴セクションと「中断を追加」ボタンを追加:

```
┌──────────────────────────────────────────┐
│ 📌 機能A — PG                            │
│ 担当: 田中  工数: 40h  期間: 4/7〜4/15   │
│                                          │
│ ── 中断履歴 ─────────────────────         │
│ (なし)                                    │
│                                          │
│ [+ 中断を追加]  [編集]  [削除]            │
└──────────────────────────────────────────┘
```

中断がある場合:

```
│ ── 中断履歴 ─────────────────────         │
│                                          │
│ ✂ 4/9 中断 — 緊急修正対応                │
│   消化: 24h → 差し込み: 緊急修正 PG 16h  │
│   [編集]  [中断を取り消し]                │
```

### 4.4 中断モーダル

```
┌──────────────────────────────────────────────┐
│ ✂ 作業中断 — 機能A PG (田中)                │
│                                              │
│ 中断日:                                      │
│ [2026-04-09 ▼]  ← カレンダーピッカー         │
│                                              │
│ 中断時点の消化工数:                           │
│ [  24  ] h / 40h  (自動: 3営業日×8h)         │
│ ↑ 手動調整可能                               │
│                                              │
│ 中断理由:                                    │
│ [緊急修正対応              ]                 │
│                                              │
│ ── 差し込み作業（任意）─────────────          │
│ □ 差し込み作業を同時に作成する               │
│ ┌────────────────────────────────┐           │
│ │ 版数:   [v1.0         ▼]      │           │
│ │ 対応名: [緊急修正     ▼] 新規入力可       │
│ │ 工程:   [PG           ▼]      │           │
│ │ 工数:   [  16  ] h            │           │
│ └────────────────────────────────┘           │
│                                              │
│            [キャンセル]  [影響を確認 →]       │
└──────────────────────────────────────────────┘
```

- 中断日: splitDateの初期値。コンテキストメニューからの場合はクリック位置の日付がプリセット
- 消化工数: 自動計算値がプリセットされるが手動調整可能
- 差し込み作業: チェックボックスで展開。版数・対応名は既存データからの選択 or 新規入力
- 「影響を確認」で影響プレビューダイアログへ遷移

### 4.5 影響プレビュー・確認ダイアログ

```
┌──────────────────────────────────────────────┐
│ 📊 影響プレビュー                            │
│                                              │
│ 以下のスケジュールが変更されます:             │
│                                              │
│ ✂ 機能A PG (田中)                           │
│   前半: 4/7〜4/9 (24h)                       │
│   後半: 4/14〜4/18 (16h)                     │
│                                              │
│ 📌 緊急修正 PG (田中) ← 新規作成            │
│   4/10〜4/11 (16h)                           │
│                                              │
│ → 機能A PT (田中) 4/16→4/21 にずれます      │
│ → 機能A IT (佐藤) 4/21→4/24 にずれます      │
│                                              │
│     [← 戻る]  [キャンセル]  [適用する]       │
└──────────────────────────────────────────────┘
```

- `analyzeImpact()` の結果を表示
- 影響がないスケジュールは表示しない
- 「戻る」で中断モーダルに戻って修正可能
- 「適用する」で `addInterruption()` + `cascadeShift()` を実行

## 5. Phase 2: ドラッグ&ドロップ差し込み（後日実装）

Phase 1の動作検証後に追加する。

### 概要

新規スケジュールバーを既存バーの上にドロップすると、ドロップ位置で自動的に既存バーを分割し、差し込みスケジュールの工数分だけ後続をずらす。

### 操作フロー

1. 新規スケジュールバーをドラッグ開始
2. 既存バーの上にホバーすると、分割プレビューが表示される
3. ドロップすると中断モーダル（もしくは確認ダイアログ）が表示される
4. 確認後に分割＋差し込みが実行される

### 技術的考慮事項

- 既存のバー移動（水平ドラッグ）と衝突しないよう、ドラッグ元の判別が必要
- Canvas上の当たり判定（既存バーの上にドロップされたかの検出）
- Phase 1のコアロジック（`addInterruption`, `cascadeShift`, `analyzeImpact`）をそのまま流用

## 6. 影響を受ける既存コード

### 変更が必要なファイル

| ファイル | 変更内容 |
|---------|---------|
| `js/schedule.js` | `addInterruption`, `removeInterruption`, `cascadeShift`, `analyzeImpact`, `calculateSegments` 追加。`calculateEndDate` の中断対応 |
| `js/schedule-render.js` | `drawScheduleBar` の分割描画対応。✂マーク・点線描画。セグメント別 `scheduleRects` 登録。コンテキストメニュー追加 |
| `js/state.js` | `interruptions` フィールドの初期値（既存データの後方互換: `interruptions` 未定義なら `[]` として扱う） |
| `js/storage.js` | `interruptions` の保存・読み込み対応（既存データマイグレーション不要: undefinedは空配列扱い） |
| `index.html` | 中断モーダル、影響プレビューダイアログのHTML追加 |
| `style.css` | 中断モーダル、コンテキストメニューのスタイル |

### 後方互換性

- `interruptions` フィールドが未定義のスケジュールは中断なし（`[]`）として扱う
- 既存データのマイグレーションは不要
- 既存の描画・操作は一切影響を受けない（`interruptions` が空なら従来通り）

## 7. 設計判断の記録

| # | 質問 | 決定 | 理由 |
|---|------|------|------|
| 1 | 対応するパターン | 割り込み型＋計画変更型の両方 | 実務では両方のケースが発生する |
| 2 | 操作方法 | 3種類すべて（コンテキストメニュー、D&D、モーダル） | ユーザーの好みやシーンに合わせた操作を提供。D&DはPhase 2 |
| 3 | 分割バーの見た目 | 点線接続（案A）＋ ✂マーク（案B）の組み合わせ | 関連が視覚的に明確かつ分割点も明示 |
| 4 | 自動ずらし範囲 | 依存チェーンを連鎖的にずらす（案②）＋確認ダイアログ | 依存関係の整合性を維持。影響が大きい場合はプレビューで確認 |
| 5 | データ構造 | 1レコードにinterruptions配列を追加（案A） | 1作業=1レコードの一貫性を保ち、中断の追加・削除が容易 |
| 6 | 分割時の工数 | 営業日自動計算＋手動調整可（案C） | 操作の簡便さと正確性のバランス |
| 7 | 差し込みスケジュール | 中断操作と同時に作成（案A）、ただし任意 | 1回の操作で完結しつつ柔軟性も確保 |
| 8 | 実装アプローチ | 段階的実装（アプローチB） | Phase 1で実用的に使える。D&Dのリスクを分離 |
