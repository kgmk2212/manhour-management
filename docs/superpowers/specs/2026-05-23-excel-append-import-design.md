---

## 1. 背景と目的

### 現状

- `js/storage.js` の `exportToExcel()` で見積・実績を Excel に出力できる
- `handleFileImport()` は JSON / Excel を拡張子で routing するが、**Excel インポート本体は未実装**（`window.handleExcelImport` が undefined）
- JSON バックアップ復元は「全データ上書き」のみ。**「既存データを残したまま追加」する経路がない**

### 目的

Excel ファイルから見積・実績を**現在のデータに追加**できる機能を提供する。既存データと競合する行は、ユーザーが行単位で「上書き / スキップ」を選択して取り込む。

### 非目標

- Excel での「全入れ替え」モードは実装しない（JSON バックアップ復元で代替）
- 設定・スケジュール・休暇など、Excel に出力されないデータ種別は本機能のスコープ外
- 不正フォーマットの自動修復・UI 上での値編集は行わない（メッセージ表示で Excel 側の修正を促す）

## 2. 確定事項サマリ

| # | 論点 | 決定 |
|---|------|------|
| 1 | スコープ | 見積・実績の両方をサポート。シート単位で取り込み対象を選択可能 |
| 2 | 見積の競合判定キー | `(版数, 対応名, 工程, 担当)` 一致 |
| 3 | 実績の競合判定キー | `(日付, 担当, 版数, 対応名, 工程, 工数)` の完全一致のみ |

| 7 | バリデーション | 寛容モード（列名マッチング、軽い類義語吸収） |
| 7-2 | 欠落値の救済 | UI 上での補完はせず、具体的なエラーメッセージで Excel 修正を促す |
| 8 | 入口 UI 配置 | 設定画面の「Excel の入出力」セクションに出力・読み込みボタンを並列配置 |

## 3. データモデルと競合判定

### 見積（estimates）

エクスポート列構成：

```
版数 / 対応名 / 工程 / 担当 / 見積工数(h) / 作業月
```

データ構造：

```js
{ id, version, task, process, member, hours, workMonths }
```

**競合判定キー**: `(version, task, process, member)` の完全一致

**競合時の挙動**:
- 上書き: `hours` と `workMonths` を読み込み側の値で更新（`id` は保持）
- スキップ: 既存を維持、読み込み側は破棄

### 実績（actuals）

エクスポート列構成：

```
日付 / 版数 / 対応名 / 工程 / 担当 / 工数(h)
```

データ構造：

```js
{ id, date, version, task, process, member, hours }
```

**競合判定キー**: `(date, member, version, task, process, hours)` の**完全一致**

理由：実績は同じ `(date, member, version, task, process)` で複数件存在することが正常運用（午前 2h + 午後 2h など合算前提）。`hours` まで完全一致した場合だけを「同ファイルの二重取り込み」と判定する。

**競合時の挙動**:
- 完全一致 → スキップ（プレビューに件数のみ表示、個別選択は不要）
- それ以外 → 常に新規追加

### バリデーション方針（X = メッセージのみ）

- 必須列が欠落 → 取り込み中止、エラー表示で欠落列名を明示
- 列名は厳密一致だが、軽い類義語（`担当者`/`担当者名` → `担当`、`hours` / `工数` / `工数(h)` の揺れ）は寛容に正規化
- セルの型不正（数値であるべき箇所が文字列など）→ 該当行は「無効」として取り込み対象外、プレビューに件数と先頭数行を表示
- 空行 → 黙ってスキップ（エラーにしない）

## 4. UI 設計

### 4.1 入口（設定タブ）

セクション名を「📊 Excel 出力」→ **「📊 Excel の入出力」** にリネームし、ボタン 2 つを並列配置：

```
📊 Excel の入出力
  実績データと見積データを Excel ファイルでやり取りします。

  [📥 Excel ファイルをダウンロード]
    現在の実績・見積データを Excel に書き出します

  [📂 Excel ファイルを読み込んで追加]
    Excel から見積・実績を現在のデータに追加します
```

「読み込んで追加」ボタンクリックで `<input type="file" accept=".xlsx,.xls">` を発火し、ファイル選択後に確認モーダルを開く。

### 4.2 確認モーダル（モックアップ案 D）

参照: `mockups/excel-append-import/mockup-d-summary-compare.html`

#### 構成

1. **ヘッダー**
   - タイトル: 「取り込み内容の確認」
   - サブ: 「取り込み元 `<ファイル名>`」
   - 右上閉じるボタン

2. **全体サマリカード**（上部・常時表示）
   - 反映予定 / 追加 / 上書き / スキップ の 4 値
   - 一括トグルや個別選択を変えると数値がリアルタイム更新

3. **シートサマリチップ**（取り込み対象選択を兼ねる）
   - 各チップにチェックボックス内蔵
   - チェック ON: 「+8 追加 / ⚠ 4 競合 / 2 重複スキップ」等の件数表示
   - チェック OFF: チップがグレーアウト、サマリとプレビューから自動的に除外

4. **見積シート: 競合 N 件の処理を選択**
   - 一括トグル: 「全て上書き / 全てスキップ / 個別判断」（デフォルト: 全て上書き）
   - 競合行を**並列比較カード**で表示（既存 vs 読み込み、変更フィールドはアンバー色で強調）
   - 各カードに行単位の「上書き / スキップ」セグメントトグル
   - 一括トグル変更で全行が同期、その後 1 行だけ個別変更すると一括トグルが自動的に「個別判断」に切り替わる

5. **実績シート**
   - 競合行（完全一致のスキップ件数）は件数のみ表示、行単位選択 UI は出さない
   - 「重複スキップ N 件」をクリック / ホバーで先頭数行を確認できる折り畳み（v1 では省略可）

6. **フッター**
   - 左: 「この内容で **39 件** を反映します」
   - 右: 「キャンセル」「取り込む（39件）」

#### 用語ルール

| 用語 | 用途 |
|------|------|
| 読み込み | import（「インポート」は使わない） |
| 取り込み | データを反映する動作（「取り込む」「取り込み内容」） |
| 追加 | 新規行の挿入 |
| 上書き | 既存行の更新（hours / workMonths を読み込み側で置換） |
| スキップ | 既存を維持、読み込み側を破棄 |
| 既存 / 読み込み | 並列比較カードの左 / 右ラベル |
| 確認してください | エラーメッセージの語尾（「修正してください」より柔らかい） |

#### バリデーション失敗時の表示

```
⚠ Excel ファイルを確認してください

見積シートの必須列が不足しています:
  ・「担当」列が見つかりません

[キャンセル] [別のファイルを選ぶ]
```

セル単位の型不正がある場合は、有効行のみ取り込み対象としつつ、プレビュー上部に：

```
⚠ 取り込めない行が 3 件あります（数値の入っていない工数など）
  3 行目: 工数列が空欄
  7 行目: 工数列が「未定」（数値でない）
  ほか 1 件
```

を表示する。

## 5. 内部設計

### 5.1 モジュール分割

`js/storage.js` が肥大化しているため、Excel インポート関連は新規モジュールに切り出す：

```
js/excel-import.js     ← 新規。本機能の本体
```

エクスポートする主な関数：

```js
// パブリック API
export async function handleExcelImport(file)  // window 経由で storage.js から呼ばれる

// 内部: パース + バリデーション
parseWorkbook(file) -> { estimateRows: [], actualRows: [], errors: [] }

// 内部: 競合検出
detectEstimateConflicts(rows, existing) -> { newRows: [], conflicts: [] }
detectActualConflicts(rows, existing) -> { newRows: [], duplicates: [] }

// 内部: モーダル UI 構築
openImportPreviewModal(parseResult)

// 内部: 確定処理
applyImport(decisions) -> 競合の選択結果を見て estimates/actuals に反映
```

### 5.2 既存コードへの影響

- `js/storage.js::handleFileImport()` の Excel 分岐: `window.handleExcelImport(file)` から `import('./excel-import.js').then(m => m.handleExcelImport(file))` に変更（dynamic import で xlsx.mjs を遅延ロード）
- `index.html` 設定タブ: 「📊 Excel 出力」セクションをリネーム + 新ボタン追加
- `js/state.js` に状態追加なし（取り込みは完了時に確定して通常の `saveData()` を呼ぶだけ）

### 5.3 取り込み確定時の処理フロー

```
1. ユーザーが「取り込む」をクリック
2. シートチェック・行ごとの選択結果を集約して decisions オブジェクト化
3. estimates 配列に対して:
   - 追加対象: 新規 id を付与して push
   - 上書き対象: 既存要素を findIndex で特定し hours / workMonths を更新
4. actuals 配列に対して:
   - 追加対象: 新規 id を付与して push
   - 完全一致のスキップ対象は何もしない
5. saveData() 呼び出し（autoBackup あり）
6. アクション履歴 (pushAction) に 'importExcel' タイプとして登録（undo 用）
7. 関連 view を再描画（renderEstimateList, renderActualList, updateReport 等）
8. トースト「39 件を取り込みました」
```

### 5.4 undo 対応

`pushAction` に登録するペイロード：

```js
{
  type: 'importExcel',
  data: {
    addedEstimates: [...],      // 追加した estimate オブジェクト群
    overwrittenEstimates: [...], // { before, after } のペア
    addedActuals: [...]
  }
}
```

undo 時：
- 追加分は id でフィルタして除去
- 上書き分は before に戻す

## 6. エッジケース

| ケース | 挙動 |
|--------|------|
| Excel が空（行 0） | 「取り込めるデータがありません」表示で終了 |
| シート名が想定外（例: `Sheet1` のみ、`実績` / `見積` が無い） | 列名から内容を推測しようとはしない。「見積シート / 実績シートが見つかりません」エラー |
| 見積と実績の両方とも 0 行有効 | サマリ「反映予定 0 件」、確定ボタン無効化 |
| 同じファイルを 2 度連続で取り込む | 見積は全件競合 → 一括トグル「全てスキップ」で何も起きない、実績は完全一致で自動スキップされる |
| `xlsx.mjs` の読み込み失敗 | アラート「Excel ライブラリの読み込みに失敗しました」表示で中止 |
| 数千行の Excel | 競合検出は O(n*m) で許容範囲。プレビューが極端に縦長になるケースは想定範囲内（仮想スクロールは初版では入れない） |
| `version` が空の見積行 | バリデーションで「無効行」扱い |
| 旧形式の `workMonth`（単数）と新形式 `workMonths`（複数）が混在 | エクスポート時に既に `workMonths.join(',')` で書き出しているため、読み込み時もカンマ分割で配列化 |

## 8. テスト観点

- 純粋関数（バリデーション・競合検出）を中心に手動テストケースを用意
  - 列名揺れ吸収（`担当` / `担当者` / `担当者名`）
  - 必須列欠落
  - 型不正セル
  - 見積の重複検出
  - 実績の完全一致重複検出
- 統合テスト相当（手動）
  - 出力 → 同ファイル取り込み → 何も変わらないこと
  - 出力 → Excel で `hours` だけ書き換え → 取り込み時に競合表示・上書き確定で値が更新されること
  - シートチェックを片方 OFF → そのシート分のみスキップされること
  - undo で取り込みを全消し
