## 1. 背景・目的

### 現状
- レポート機能には静的分析（ルールベース）が実装済みで、割と正確
- インサイト、異常検知、警告タスク、工程精度分析、キャパシティ分析などがある

### 目的
- 既存の静的分析は**そのまま残す**
- LLM によって以下を**追加**する:
  - 将来の予測
  - 今後の方針提案
  - 推奨されるアクション（特に開発プロセス改善による工数削減・効率向上）

### ターゲットユーザー
- **当面は 1 人（開発者本人）**
- マルチチーム運用は将来の拡張として棚上げ（§4 参照）

## 2. アーキテクチャ検討

### 2.1 前提条件（2026-04-20 更新）

- アプリは **GitHub Pages** で配信（パブリック静的ホスティング）
- データはブラウザの **localStorage** に保存
- LLM は **開発者ローカルマシン**の Ollama（`localhost:11434`）
- GPU サーバーや共有フォルダ、マルチチーム運用は**存在しない**

### 2.2 採用方針（2026-04-20 更新）: ブラウザ直叩き方式

旧方針「バッチ推論 + 結果キャッシュ（LLM サーバー + cron + 共有フォルダ）」を廃止し、**ブラウザから直接 Ollama に fetch する**方式に変更する。

```
┌─────────────────────────────────────────┐
│  開発者マシン (ローカル)                   │
│                                         │
│  ┌──────────────────┐                   │
│  │ ブラウザ           │                   │
│  │  (GitHub Pages上  │                   │
│  │   のアプリ)        │                   │
│  │                  │                   │
│  │  localStorage    │                   │
│  │  → 要約(JS)      │                   │
│  │  → fetch POST ───┼──▶ localhost      │
│  │                  │    :11434         │
│  │  ◀─ 結果JSON ────┼──── Ollama        │
│  │                  │   (qwen3.5:9b 等) │
│  └──────────────────┘                   │
└─────────────────────────────────────────┘
```

### 2.3 選定理由

- **実質 1 人運用**なので、旧プランAの却下理由「各 PC に Ollama 導入が非現実的」が消失
- 同時アクセスが無いので**リアルタイム推論でも GPU 負荷に問題なし**
- **データが手元を離れない**（パブリック GitHub Pages に乗っていても、LLM 推論は localhost 完結）
- **サーバー基盤ゼロ**（cron / rsync / NAS / VPS 不要）
- 既存の Python パイプライン（`summarize.py` / `analyze.py`）のロジックを JS に移植すればよく、プロンプト資産は流用可能

### 2.4 フロントエンド側の表示

- 「分析を実行」ボタンで推論を開始 → ローディング → 結果を描画
- 結果 JSON は `localStorage` にキャッシュ（次回表示時は即時描画、再実行ボタンあり）
- `analysis/latest.json` が存在する場合もフォールバックとして読み込む（既存挙動維持）
- 生成日時・使用モデル名を明示
- 静的分析（既存）と LLM 分析は明確に区別して並列表示

### 2.5 検討した他の方式と再評価

| 方式 | 2026-02-23 時点の不採用理由 | 2026-04-20 時点の再評価 |
|------|--------------------------|----------------------|
| **プランA: Ollama 直接通信** | 全リーダーの PC 管理が非現実的 | **✅ 採用**（1 人運用で理由が消失） |
| プランB: WebLLM | WebGPU 要件、モデルサイズ制限 | 依然として品質で劣る。qwen3.5:9b のほうが安定 |
| プランC: リアルタイム API (共有 LLM サーバー) | 同時アクセスで GPU 負荷集中 | サーバー基盤維持コストに見合わない（1 人運用） |
| バッチ推論 + 結果キャッシュ（旧採用） | — | cron / 共有フォルダ / 別 GPU サーバーが前提で、インフラが無いので実現不能 |
| Cloudflare Pages + Workers AI | — | 無料枠ありだがモデル品質が qwen に劣る。データ外出し。マルチユーザ運用に戻す日まで保留 |
| Vercel + Anthropic/OpenAI API | — | 品質最高だが有料・データ外出し。業務データには不適 |
| **E1: トンネル変種 (Cloudflare Tunnel / ngrok)** | — | **併用候補**として採用可。クラウドデプロイ + 自宅 Ollama を外部からも使えるようにする。詳細は [アーキテクチャ 付録 B](./LLM_ANALYSIS_ARCHITECTURE.md#付録-b-トンネル変種-e1) |

### 2.6 CORS 対応（唯一の設定作業）

GitHub Pages (HTTPS) から `http://localhost:11434` を叩くため、Ollama 側で以下を設定する:

```bash
# macOS (launchctl で Ollama を常駐させている場合)
launchctl setenv OLLAMA_ORIGINS "https://kgmk2212.github.io"
# Ollama を再起動

# 一時起動時
OLLAMA_ORIGINS="https://kgmk2212.github.io" ollama serve
```

Mixed Content ブロック回避のため、`http://` を `https://` のページから叩くのは制約になる。後述の通り、localhost は例外扱いされるためブラウザは許可する（実機検証は Phase 3 で行う）。

## 3. LLM に推論させる内容

### 方針
- **開発プロセスの改善 → 工数削減・効率向上**に繋がる提案を出す
- 「何が起きているか」は既存の静的分析がカバー済み
- LLM は「**なぜ**起きているか、**どうすれば**改善できるか」を担う

### 分析カテゴリ

| カテゴリ | 入力データ | 推論内容 |
|---------|----------|---------|
| 将来予測 | 過去の見積vs実績の傾向 | 完了見込み時期（楽観/現実/悲観） |
| プロセス改善提案 | メンバー別の工数・精度データ | 見積手法の見直し、レビュープロセス改善等 |
| リスク警告 | 進捗率・残工数・キャパシティ | 遅延傾向の指摘と具体的な対策 |
| 振り返り分析 | 完了バージョンの全データ | 見積精度の評価、改善ポイントの指摘 |
| 最適配置提案 | メンバー稼働状況＋今後の予定 | リソース配分の最適化提案 |

### 提案の方向性（例）

| 静的分析が示す事実 | LLM が出すべき提案 |
|---|---|
| テスト工程の見積精度が低い | 見積手法の見直し（過去類似案件との比較、チェックリスト導入等） |
| 特定メンバーの実績が見積を常に超過 | タスク分割の粒度見直し、ペアワーク導入、スキル補強 |
| ある工程でいつも手戻りが発生 | レビュープロセスの前倒し、完了定義の明確化 |
| 月末にタスクが集中する傾向 | スプリント計画の見直し、WIP 制限の導入 |

## 4. データ連携（新方針で解消）

### 核心的な問題（旧）
> 各ユーザーのデータが localStorage にある → LLM サーバーにどう集めるか

### 新方針での解消

**1 人運用 + ブラウザ直叩き**により、この問題自体が消滅する:

- localStorage → ブラウザ内で要約 → ブラウザから Ollama へ POST
- データは一度もマシンを離れない
- 共有フォルダ / rsync / teamId / 運用周知は**すべて不要**

### 将来マルチユーザ運用に戻す場合

下記のいずれかを復活させる必要がある。現時点では**棚上げ**:

| 方法 | 備考 |
|------|------|
| 共有 LLM サーバー (原案) | 社内 GPU サーバーが用意できた時点で検討 |
| Cloudflare Workers AI + D1 | マルチユーザ化 + データ集約をクラウドで完結させる案 |
| localStorage → Supabase 等への移行 | アプリ本体のデータ永続化先を変更する根本策 |

## 5. データ連携フロー（新方針）

```
ブラウザ (GitHub Pages 上のアプリ)
    │
    │ 1. ユーザー操作「分析を実行」ボタン押下
    ▼
要約生成 (JS)
    │ localStorage から読み出し
    │ summarize.py 相当のロジックで集計
    │ → 要約 JSON を生成（in-memory）
    ▼
Ollama 呼び出し (JS fetch)
    │ POST http://localhost:11434/api/chat
    │ system/user プロンプトは既存資産を流用
    ▼
結果 JSON
    │ スキーマ検証
    │ localStorage にキャッシュ保存
    ▼
AI 分析セクションを描画 (既存 js/ai-analysis.js を拡張)
```

## 6. プロンプト設計

実装済みの資産:

- `llm-analysis/prompts/system_prompt.txt` — 共通システムプロンプト
- `llm-analysis/prompts/system_prompt.gemma4.txt` — gemma4 用（chat + prefill）
- `llm-analysis/prompts/system_prompt.qwen3.5.txt` — qwen3.5 用（generate）
- `llm-analysis/prompts/output_format.txt` — JSON 出力スキーマ

### 6.1 設計方針（継続）

- **生データではなく要約データを渡す**: トークン節約、小モデルでの動作
- **JSON 出力を指定**: フロントエンドで構造化表示しやすい
- **「データ上の根拠」を必須に**: ハルシネーション抑制

### 6.2 入力となる要約 JSON の構造

実装は `llm-analysis/summarize.py` を参照。主要フィールド:

```
overall           — 全体の見積/実績/精度
by_version        — バージョン別の進捗・精度・最大超過タスク
by_process        — UI/PG/PT/IT/ST の精度
by_member         — メンバー別 + 得意/苦手工程
member_monthly    — メンバー × 月の工数（負荷集中検出）
monthly_trend     — 月次の見積・実績・精度
task_sizes        — タスクサイズ分布（大タスク検出）
capacity          — キャパシティ警告月
anomalies         — 超過率 50% 以上のタスク
```

### 6.3 出力 JSON スキーマ

実装は `llm-analysis/analyze.py` の `OUTPUT_SCHEMA` を参照。トップレベル:

```
meta                  — 生成日時 / モデル / 分析期間
team_evaluation       — 総合評価（headline, summary, score A-E, strengths, weaknesses, trend, analysis）
outlook               — 展望（version_forecasts, accuracy_forecast, resource_risks）
recommended_actions   — 推奨アクション配列（priority, category, title, rationale, action_summary, action_steps, expected_effect）
next_review_focus     — 次回の注視ポイント
```

## 7. 技術的な検討事項

### モデル選択（Phase 1 検証結果）

- 候補を `qwen3.5:9b` と `gemma4` で比較（`llm-analysis/data/results/` に試行ログ）
- 運用モデルは後続 Phase で確定

### セキュリティ

- ローカル LLM（`localhost:11434`）を叩くため、**データは外部に送信されない**
- GitHub Pages から localhost への POST は、ブラウザが localhost を特別扱いするため Mixed Content で止められない想定（Phase 3 で実機確認）

### 実装上の制約

| 制約 | 対応 |
|------|------|
| Ollama が落ちていると動かない | 未起動時は「Ollama を起動してください」と案内、`analysis/latest.json` があればフォールバック表示 |
| JSON 出力が不安定（小モデル） | 既存の retry/検証ロジックを JS 側にも移植 |
| 推論に 30 秒〜数分かかる | 進捗表示（ストリーミング or スピナー）、キャンセルボタン |
