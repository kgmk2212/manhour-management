# ローカルLLMによるレポート分析機能 - 構想メモ

> **ステータス**: 構想段階（2026-02-23時点）
> **次のステップ**: 1人分のデータでLLM出力品質を検証

---

## 1. 背景・目的

### 現状
- レポート機能には静的分析（ルールベース）が実装済みで、割と正確
- インサイト、異常検知、警告タスク、工程精度分析、キャパシティ分析などがある

### 目的
- 既存の静的分析は**そのまま残す**
- LLMによって以下を**追加**する：
  - 将来の予測
  - 今後の方針提案
  - 推奨されるアクション（特に開発プロセス改善による工数削減・効率向上）

### ターゲットユーザー
- **チームリーダー**（複数人が同時利用する）

---

## 2. アーキテクチャ検討

### 前提条件
- アプリは社内ローカルサーバーで配信（静的ファイル配信）
- 各ユーザーのデータはブラウザのlocalStorageに保存
- LLMサーバーはアプリサーバーとは別のサーバー（GPU搭載、ただし高性能ではない）

### 採用方針: バッチ推論 + 結果キャッシュ方式

#### 選定理由
- 複数チームリーダーが同時利用する → 各PCにOllamaは非現実的
- 同時アクセスがある → リアルタイム推論はGPU負荷が厳しい
- 工数データの更新頻度は低い → 日次/週次のバッチで十分
- GPUが控えめ → 夜間に1件ずつゆっくり推論すればいい

#### 全体構成

```
各チームリーダーのPC (ブラウザ)
    │  データはlocalStorageに保存
    │
    ▼
社内ローカルサーバー（アプリサーバー）
    ├── index.html / style.css / js/*.js  ← 既存アプリ
    └── analysis/*.json                   ← LLM推論結果を配置
                    ▲
                    │ 定期コピー
                    │
LLMサーバー（別マシン・GPU搭載）
    ├── データ取り込み → LLM推論 → 結果JSON生成
    └── Ollama等でローカルLLMを実行
```

#### フロントエンド側の表示
- 結果JSONが存在すれば `fetch` して表示
- 存在しなければセクション自体を非表示（graceful degradation）
- 生成日時を明示（「分析日: 2026-02-23」等）
- 静的分析（既存）とLLM分析は明確に区別して並列表示

### 検討した他の方式

| 方式 | 概要 | 不採用理由 |
|------|------|-----------|
| プランA: Ollama直接通信 | 各PCでOllamaを起動しブラウザから直接通信 | 全リーダーのPCにインストール・管理が非現実的 |
| プランB: WebLLM | ブラウザ内WebGPUでLLM実行 | WebGPU対応要件、モデルサイズ制限 |
| プランC: リアルタイムAPI | LLMサーバーにリアルタイムで問い合わせ | 同時アクセスでGPU負荷集中、待ち時間発生 |

## 3. LLMに推論させる内容

### 方針
- チームリーダー向けに、**開発プロセスの改善 → 工数削減・効率向上**に繋がる提案を出す
- 「何が起きているか」は既存の静的分析がカバー済み
- LLMは「**なぜ**起きているか、**どうすれば**改善できるか」を担う

### 分析カテゴリ

| カテゴリ | 入力データ | 推論内容 | 更新頻度 |
|---------|----------|---------|---------|
| 将来予測 | 過去の見積vs実績の傾向 | 完了見込み時期の予測 | 週1回 |
| プロセス改善提案 | メンバー別の工数・精度データ | 見積手法の見直し、レビュープロセス改善等 | 月1回 |
| リスク警告 | 進捗率・残工数・キャパシティ | 遅延傾向の指摘と具体的な対策 | 週1回 |
| 振り返り分析 | 完了バージョンの全データ | 見積精度の評価、改善ポイントの指摘 | バージョン完了時 |
| 最適配置提案 | メンバー稼働状況+今後の予定 | リソース配分の最適化提案 | 月1回 |

### 提案の方向性（例）

| 静的分析が示す事実 | LLMが出すべき提案 |
|---|---|
| テスト工程の見積精度が低い | 見積手法の見直し（過去類似案件との比較、チェックリスト導入等） |
| 特定メンバーの実績が見積を常に超過 | タスク分割の粒度見直し、ペアワーク導入、スキル補強 |
| ある工程でいつも手戻りが発生 | レビュープロセスの前倒し、完了定義の明確化 |
| 月末にタスクが集中する傾向 | スプリント計画の見直し、WIP制限の導入 |

## 4. データ連携の課題（未解決）

### 核心的な問題
各ユーザーのデータがlocalStorageにある → LLMサーバーにどう集めるか

### 検討中の方法

| 方法 | 概要 | メリット | デメリット |
|------|------|---------|-----------|
| A. 手動/半自動エクスポート | バックアップ機能でエクスポート → 共有フォルダ → LLMサーバーにコピー | シンプル | 人に依存する |
| B. アプリから自動送信 | フロントエンドからPOSTでデータ送信 | ユーザーの手間ゼロ | サーバーにAPI受け口が必要（静的配信では済まない） |
| C. データ保存先の変更 | localStorageをやめてサーバーにデータ集約 | 根本的解決 | アプリの設計を大きく変える |

### 所見
- LLM分析を複数チームで本格運用するなら、遅かれ早かれ**データの集約**が必要
- 「全チームの横断レポート」等の要件でも同じ課題が発生する
- ただし、まずはLLM出力の品質検証が先

---

## 5. データ連携フロー（暫定案）

バックアップファイルを起点とする方式：

```
各チームリーダーのPC
    │ バックアップ機能でJSON出力
    ▼
バックアップフォルダ（共有フォルダ等）
    │ 自動コピー or 定期実行 (rsync/scp等)
    ▼
LLMサーバー
    │ バックアップJSONを読み込み → 要約データ生成 → LLM推論
    │ 推論結果JSONを出力
    ▼
結果ファイル
    │ 自動コピー or 定期実行
    ▼
アプリサーバー (analysis/*.json として配置)
    │ フロントエンドが fetch して表示
    ▼
各チームリーダーのブラウザ
```

## 6. プロンプト設計案（現時点での案）

### 6.1 設計方針

- **生データではなく要約データを渡す**: トークン節約、計算精度の確保、小モデルでの動作を考慮
- **要約JSONの生成**: バッチスクリプト（Python等）でバックアップJSONから集計。既存の `report.js` の分析ロジック（`renderInsights`, `renderAnomalyDetection`, `renderMemberPerformance` 等）とほぼ同じ計算なので移植は容易
- **JSON出力を指定**: フロントエンドで構造化表示しやすい
- **「データ上の根拠」を必須に**: ハルシネーション抑制

### 6.3 システムプロンプト

```
あなたはソフトウェア開発チームの工数分析アドバイザーです。
チームリーダーに対して、チームの評価と今後の展望を報告してください。

## あなたの役割
- 客観的なデータに基づいてチームの現状を評価する
- 将来のリスクと機会を予測する
- 開発プロセスの改善によって工数削減・効率向上につながる具体的なアクションを提案する
- チームリーダーが意思決定に使える情報を提供する

## 分析の観点

### 1. チーム総合評価
以下の観点でチームの現状を評価してください。
- 見積精度の全体的な傾向（改善/悪化/安定）
- 工程別の強み・弱み
- メンバーの稼働バランス
- キャパシティの適正度

### 2. 展望と予測
データの傾向から以下を推測してください。
- 進行中バージョンの完了見込み（楽観/現実的/悲観）
- 見積精度の今後の変化予測
- リソース面のリスク

### 3. 推奨アクション
開発プロセスの改善提案を具体的に出してください。
チームの習慣やプロセスを変えることで改善できるものを優先してください。
以下のカテゴリから、データが示す課題に合った提案をしてください。

- 見積プロセスの改善（例: 過去実績ベースの見積、バッファの設定方法、チェックリスト導入）
- レビュー・品質プロセス（例: 工程間レビューの追加、完了定義の明確化）
- タスク管理（例: タスク分割の粒度、WIP制限、優先度の見直し）
- リソース配分（例: メンバーの得意工程への集中、ペアワーク、スキルトランスファー）
- 計画・スケジューリング（例: バッファの積み方、マイルストーン設定）

各アクションには以下を含めてください：
- なぜそれが必要か（データ上の根拠）
- 具体的に何をするか
- 期待される効果

## 制約
- 数値データに基づかない推測は避けてください
- 提案は3〜5個に絞り、優先度をつけてください
- チームリーダーが明日から実行できる粒度の提案にしてください
- 日本語で回答してください
```

### 6.5 設計上の判断とその理由

| 判断 | 理由 |
|------|------|
| 生データではなく要約を渡す | トークン節約、計算精度の確保、小モデルでの動作 |

| next_review_focus を入れる | 次回分析時の追跡ポイントを明示 |

## 9. 技術的な検討事項

### モデル選択
- 7B〜14Bクラスで十分（数値データの要約・パターン認識が主）
- 具体的なモデルは検証時に選定

### GPU負荷の軽減策
- プロンプトを工夫して短い構造化出力（JSON）にする
- 分析を分割実行（「予測」「リスク」「提案」を別々に推論）
- 差分更新（前回から変化があったデータだけ再分析）
